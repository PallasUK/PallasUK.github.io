What is DevSecOps?

The DevOps movement was not started from a single manifesto, instead it grew from a desire to to break down the barriers between development and operations staff.

This means having operations staff supporting development of the product and involving development staff in resolving production issues. This allows the reuse of deployment and software configuration effort and means software is developed for the production environment.

The increased collaboration allows teams to quickly develop and deploy new features. Allowing minimum viable products to be delivered and iterated upon. This reduces the risk the software will fail to meet the needs of the product owner.

However it isn't enough to deploy an application quickly, you need to ensure the environment is secure and the application isn't exposing any vulnerabilities. Traditional development would place reviews with quality assurance and information security before moving to production. With the faster pace of change these teams need to be incorporated into the process and involved from design up to deployment. This has lead to the term DevSecOps.

What can we do to help?

We will try to reuse DevSecOps tools, scripts and libraries accross projects allowing us to quickly stand up development teams and focus on deliverying a specific product.

We have a number of Continious Integration (CI) pipelines for various software programming languages looking to enable various assurance activities on our software. We look to automate new requirements into existing pipelines to ensure all customers can benefit from the enhancements. We have automated issue tracking, release management, automated testing and code analysis activities for various development stacks and languages.

We have developed various Continious Deployment (CD) pipelines, we have used Puppet and Ansible to deploy software to physical servers and deployed packages to serverless platforms (e.g. AWS Lambda). While we can provide full automated Ci/CD pipeliness we can also create gated processes to allow for assurance review.

Pallas collects metrics from all CI/CD pipelines, this information is used to inform development of the software. Allowing us to focus on issues appropriate for the software development approach

  • Establish delivery pipelines
  • Integration of assurance into CI/CD
  • Identifying Metrics from CI/CD to enable project tracking
  • Coaching on integration of ITIL with DevSecOps
  • Coaching on Onion and Stride security models
  • Standardisation of CI/CD pipelines to reduce technical debt
